# Rock and Lever

> [Problem on Codeforces](https://codeforces.com/problemset/problem/1420/B)

## Idea

The key insight is to focus on the **most significant bit (MSB)** of each number in the array. For two numbers $a_i$ and $a_j$ to satisfy $a_i \& a_j \geq a_i \oplus a_j$, their MSBs must be the same. This is because the MSB determines the largest power of $2$ in the binary representation, and matching MSBs ensures that the result of the bitwise AND operation is sufficiently large to satisfy the inequality. Thus, the problem reduces to counting pairs of numbers with the same MSB.

## Approach

1. **Group by MSB:** For each number $a_i$ in the array, compute the position of its most significant bit (using `__builtin_clz` or equivalent). Group numbers by their MSB position into a frequency map `mp`, where `mp[b]` counts how many numbers have their MSB at position $b$.

2. **Count valid pairs:** For each group in `mp`, compute the number of valid pairs using the combination formula $\binom{k}{2} = \frac{k \cdot (k - 1)}{2}$, where $k$ is the size of the group. This works because any two numbers in the same group will have the same MSB and thus satisfy the given inequality.

3. **Sum the results:** Add up the pair counts from all groups to get the total number of valid pairs for the array.

4. **Repeat for all test cases.**

## Complexity

- **Time:** $O(n)$ per test case, where $n$ is the size of the array. Computing the MSB for each number is $O(1)$, and summing over the groups is proportional to the number of distinct MSB values, which is bounded by $\lceil \log_2(10^9) \rceil = 30$.
- **Space:** $O(\log a_{\max}) = O(30)$ for the frequency map, where $a_{\max}$ is the largest number in the array.

## Notes

- The use of `__builtin_clz` (count leading zeros) is efficient for determining the MSB position. It avoids explicitly calculating logarithms and works directly with the binary representation of the number.
- The constraint that the sum of all $n$ across test cases does not exceed $10^5$ ensures that the algorithm runs efficiently within the given limits.
